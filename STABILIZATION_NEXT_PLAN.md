# AI-cutting 稳定化下一步规划

## 1. Summary

当前 AI-cutting 已经超过最初的 Web MVP：上传、火山 ASR、自动剪辑、批量任务、字幕/封面样式预设、精确预览、可选导出、任务下载这些主链路都已经具备。

下一阶段不建议继续盲目加大功能，而是先做一轮“稳定化”和“可交付化”，目标是让项目从“自己能用”变成“可以放心给自媒体朋友试用”。

本轮计划一次性完成 5 件事：

1. 清理缓存和输出管理
2. 做真实端到端回归测试
3. 固定 UI 到一个更稳定的版本
4. 新增系统自检页
5. 为后续 LLM 智能剪辑打基础，但暂不接入自动剪辑

## 2. 当前状态判断

### 2.1 已完成能力

- 本地 Web 页面
- 视频拖拽/上传
- 单视频和多视频批量处理
- 每个视频可单独填写封面标题
- 火山引擎 ASR 字幕识别
- 剪后音频识别，字幕同步比早期更准
- 静音、气口、部分重复候选检测
- 字幕和封面样式预设
- 样式精确预览
- 输出成片和封面
- 可选导出 ASS/SRT、火山 JSON、剪辑报告
- 任务详情和下载
- 一键启动脚本
- README 和基础编码检查

### 2.2 当前主要问题

- `web/static/style_previews/` 会不断产生预览图片和 ASS 文件，长期使用会越来越乱。
- 缺少系统自检页，别人拿到项目后不知道环境哪里没配好。
- 功能改动较多，需要一次完整回归，确认主流程没被 UI 或导出逻辑影响。
- UI 仍然需要定稿，避免反复推翻。
- LLM 智能剪辑已经有方向，但现在还不适合直接上自动删除内容。

## 3. Task A：缓存和输出管理

### 3.1 目标

解决预览缓存、临时文件、任务输出无限增长的问题，让项目长期使用时不会越来越臃肿。

### 3.2 需要做什么

- 新增缓存清理逻辑：
  - 清理 `web/static/style_previews/` 下过期的 `accurate_*.jpg`
  - 清理 `web/static/style_previews/` 下过期的 `accurate_*.ass`
  - 可选清理旧的抽帧预览图
- 设计保留策略：
  - 默认保留最近 100 个预览文件，或
  - 默认删除超过 24 小时的预览文件
- 新增手动清理入口：
  - 设置页添加“清理预览缓存”按钮
  - 显示清理前后的文件数量和大小
- 明确不清理：
  - `jobs/` 任务输出
  - 用户上传的视频
  - 用户生成的成片和封面
  - `settings.local.json`
  - `style_presets.local.json`

### 3.3 建议实现

- 在 `web/app.py` 新增工具函数：
  - `_preview_cache_stats()`
  - `_cleanup_preview_cache(max_age_hours=24, keep_latest=100)`
  - `_format_size()`
- 在设置页新增一个 POST 接口：
  - `POST /settings/cleanup-preview-cache`
- 设置页显示：
  - 预览缓存文件数量
  - 预览缓存总大小
  - 上次清理结果

### 3.4 验收标准

- 点击清理后，旧的 `accurate_*.jpg/.ass` 会被删除。
- 最近生成的预览仍然可用。
- 不会删除任务输出文件。
- 页面显示清理结果，例如“已清理 128 个文件，共 42.3 MB”。

## 4. Task B：真实端到端回归测试

### 4.1 目标

确认真实用户流程是通的，而不是单独接口都能跑。

### 4.2 测试流程

准备一个 30 秒到 2 分钟的短口播视频，然后测试：

1. 打开首页。
2. 上传单个视频。
3. 填写封面标题，包含换行。
4. 选择字幕样式预设。
5. 使用“标准”剪辑强度。
6. 默认只导出成片和封面。
7. 点击开始处理。
8. 等待任务完成。
9. 下载成片和封面。
10. 检查：
    - 字幕是否出现
    - 字幕是否使用所选预设
    - 封面是否使用封面标题样式
    - 输出文件是否为中文命名
    - 没勾选的额外文件是否没有生成

再测试一次批量：

1. 上传 2 个视频。
2. 检查封面标题区域是否变成逐条填写。
3. 给每条视频填写不同标题。
4. 开始处理。
5. 检查每个视频输出目录和 ZIP。

### 4.3 建议补充一个测试记录文件

新增：

```text
TEST_RUNBOOK.md
```

内容包括：

- 测试视频文件名
- 测试日期
- 使用的预设
- 是否成功
- 发现的问题
- 修复状态

### 4.4 验收标准

- 单视频任务可完整跑通。
- 批量任务可完整跑通。
- 任务失败时日志能定位原因。
- 输出文件符合当前设计：默认只生成成片和封面，额外导出按勾选生成。

## 5. Task C：UI 定稿和小修

### 5.1 目标

停止无限推翻 UI，把当前版本收敛成一个稳定可用的工具界面。

### 5.2 本轮不做

- 不继续大规模重构视觉风格。
- 不追求完全复刻剪映。
- 不引入 React/Vue。
- 不做复杂时间线编辑器。

### 5.3 本轮要做

- 检查首页：
  - 上传区是否清晰
  - 开始处理按钮是否明显
  - 多视频标题是否好理解
  - 最近任务是否干扰主流程
- 检查样式预设页：
  - 预览是否足够突出
  - 保存按钮是否始终可见
  - 字幕/封面切换是否清楚
  - 画幅切换是否清楚
  - 设置项是否过密
- 检查设置页：
  - 火山配置是否清楚
  - Token 本地保存提示是否足够
  - 新增缓存清理区是否不干扰主配置
- 检查任务页：
  - 状态是否中文
  - 输出文件是否好下载
  - 错误日志是否可读

### 5.4 具体小修方向

- 统一按钮文案。
- 统一状态文案。
- 减少不必要说明文字。
- 避免页面出现 `running/done/failed` 这类英文状态。
- 保证移动端 375px 无横向滚动。
- 保证所有页面无连续问号乱码。

### 5.5 验收标准

- `/`、`/style-presets`、`/settings`、`/jobs/{id}` 都正常打开。
- 桌面 1280x720 下核心操作可见。
- 手机 375px 下无横向滚动。
- 浏览器控制台无明显错误。
- 用户不需要读说明也能知道下一步做什么。

## 6. Task D：系统自检页

### 6.1 目标

给未来试用者一个“环境是否准备好”的页面，减少配置问题造成的困惑。

### 6.2 新增页面

建议新增：

```text
GET /health
```

导航里可叫：

```text
自检
```

### 6.3 自检项目

基础环境：

- Python 路径
- Python 版本
- 当前项目路径
- 当前工作目录

视频工具：

- `ffmpeg` 是否可用
- `ffprobe` 是否可用
- FFmpeg 是否支持 `libass`

火山配置：

- APP ID 是否存在
- Access Token 是否存在
- Token 不显示明文，只显示已配置/未配置

目录权限：

- `jobs/` 是否可写
- `web/static/style_previews/` 是否可写
- `web/settings.local.json` 是否可写
- `web/style_presets.local.json` 是否可写

磁盘状态：

- 项目所在磁盘剩余空间
- `jobs/` 占用空间
- `style_previews/` 占用空间

可选网络检查：

- 不建议默认请求火山接口，避免消耗或误触发。
- 可以只显示“凭证已配置，真实识别会在任务中验证”。

### 6.4 UI 设计

自检页应像一个检查清单：

```text
通过 / 警告 / 失败
检查项
说明
建议动作
```

示例：

```text
失败  FFmpeg 不可用
说明：系统找不到 ffmpeg 命令。
建议：安装 FFmpeg，并确保 ffmpeg 可以在 CMD 中直接运行。
```

### 6.5 验收标准

- 未安装 FFmpeg 时能明确提示。
- 未配置火山 Token 时能明确提示。
- 不泄露 Access Token。
- 能看到缓存和任务输出占用空间。
- 页面能帮助非技术用户判断下一步。

## 7. Task E：LLM 智能剪辑前置准备

### 7.1 目标

不直接接入 LLM 自动剪辑，但为后续智能剪辑做好数据基础。

### 7.2 本轮不做

- 不调用大模型。
- 不自动删除语义内容。
- 不做人工审核界面。
- 不做时间线编辑器。

### 7.3 本轮可以做的轻量准备

- 确认火山 ASR 返回里是否有 `words` 字段。
- 如果有，保留到额外导出的 JSON 里。
- 设计但不一定实现：
  - `transcript_words.json`
  - `pause_analysis.json`
  - `edit_suggestions.json`
- 在 `LLM_SMART_EDITING_PLAN.md` 里补一段“当前实现状态”。

### 7.4 后续真正进入 LLM 阶段时的路线

第一阶段只生成建议报告：

```text
原视频/剪后视频
-> ASR 逐词稿
-> 停顿分析
-> LLM 生成 edit_suggestions.json
-> 生成 review.html
```

不要一开始就自动剪。

### 7.5 验收标准

- 当前项目文档明确：LLM 是后续阶段，不是当前稳定化阶段。
- 现有导出文件能尽量保留未来智能剪辑需要的数据。
- 不影响当前火山字幕主流程。

## 11. 风险

### 11.1 缓存清理误删

风险：

误删用户还在看的预览图。

规避：

- 只清理 `accurate_*.jpg/.ass`
- 保留最近 N 个
- 不清理任务输出

### 11.2 自检页泄露 Token

风险：

把火山 Access Token 显示到页面。

规避：

- 只显示“已配置/未配置”
- 不显示 token 前后几位
- 不写入日志

### 11.3 端到端测试耗时

风险：

视频太长，调试周期慢。

规避：

- 使用 30 秒到 2 分钟短视频
- 先只测默认导出
- 再测额外导出

### 11.4 UI 再次陷入无限重构

风险：

一直觉得不够好，导致稳定化没推进。

规避：

- 本轮只做小修
- 样式页不再推翻结构
- 先把工具稳定性交付出来

## 12. Done Definition

本轮完成的标准：

- 预览缓存可以统计和清理。
- 新增 `/health` 系统自检页。
- 首页、设置页、样式页、任务页无明显乱码和横向溢出。
- 单视频真实任务跑通。
- 批量视频真实任务跑通。
- 输出文件符合勾选逻辑。
- README 更新。
- `TEST_RUNBOOK.md` 记录一次真实测试。
- 不泄露火山 Token。

## 13. 最终判断

做完这一轮后，AI-cutting 应该进入“可以给朋友试用”的状态。

之后再进入下一阶段：

```text
LLM 智能建议报告
字幕纠错
重复/废话候选标注
人工确认后半自动剪辑
```

不要在当前阶段直接让 LLM 自动删视频内容。
