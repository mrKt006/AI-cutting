# AI-cutting 阶段性开发清单

## 目标

将当前本地自动剪辑 MVP 逐步建设为：

1. 字幕、断句和自动删减结果可靠的本地专业版。
2. 能记录 AI 决策与真实用户修改数据的可评估产品。
3. 具备任务可靠性、用户隔离、密钥安全和成本控制的公网商业版。

本路线遵循以下原则：

- 音频决定用户实际说了什么，逐字稿决定文字应当怎样写。
- 除明确记录的删除操作外，任何处理阶段都不能漏字、添字或改变 token 顺序。
- 系统负责计算合法候选，AI 负责在合法候选中做语义选择。
- 自动剪辑是默认流程，精修台用于处理少量不满意结果。
- 先保证正确性、可靠性和安全性，再优化速度和规模。

## 阶段 1：修复字幕完整性与时间映射

### 目标

首先解决当前最严重的问题：AI 断句正确，但静音剪辑和时间映射后出现漏字、串句和句尾丢失。

### 开发清单

- [ ] 为原始 ASR token、明确删除 token、保留 token 和最终字幕 token 建立统一 ID 映射。
- [ ] 删除仅根据 token 中点判断是否保留的脆弱逻辑。
- [ ] token 与保留片段部分重叠时，正确裁切和映射时间，而不是直接丢弃。
- [ ] 静音剪辑不能跨越受保护的语音 token。
- [ ] 明确区分静音删除、AI 删除、人工删除和时间线裁切。
- [ ] 在断句前和生成字幕前各执行一次 token 完整性校验。
- [ ] 校验失败时停止输出错误成片，并在任务页显示具体缺失 token。
- [ ] 将映射前后 token 数量、缺失 ID 和删除原因写入诊断文件。
- [ ] 对旧任务保留兼容读取，不重写原始 ASR 数据。

### 必须满足的不变量

```text
最终保留 token + 有明确原因的删除 token = 原始修订 token
```

- [ ] 不漏字。
- [ ] 不重复 token。
- [ ] 不改变 token 顺序。
- [ ] 所有删除都有类型、时间、理由和来源。

### 测试清单

- [ ] 句尾紧邻静音时不丢失“了、事、户、角、话”等字符。
- [ ] 静音中间存在极短发音时不会误删。
- [ ] AI 删除口误后，相邻句不会错误拼接。
- [ ] 使用任务 `20260713-142415-589059` 验证已知串句全部消失。
- [ ] 预览字幕、ASS、SRT 和最终烧录文本一致。

### 阶段出口

- [ ] 除明确删除内容外，测试集中未出现任何 token 丢失。

## 阶段 2：逐字稿文件解析与标题解耦

### 目标

支持用户上传 `.md` 或 `.txt`，自动解析内容标题、封面标题和逐字稿，并让内容标题与封面标题完全独立。

格式规范见 [TRANSCRIPT_FORMAT.md](./TRANSCRIPT_FORMAT.md)。

### 解析器清单

- [ ] 支持 UTF-8 `.md`。
- [ ] 支持 UTF-8 `.txt`。
- [ ] Markdown 支持 `内容标题`、`封面标题`、`逐字稿`。
- [ ] 兼容 `最终文案` 和 `原始文案`，按规范确定优先级。
- [ ] 识别并忽略 `[待确认]`、`[暂无]` 等占位内容。
- [ ] TXT 支持中文方括号板块标记。
- [ ] 未知章节忽略并产生非阻塞提示。
- [ ] 重复关键章节时阻止提交并明确指出冲突位置。
- [ ] 板块外零散内容不混入逐字稿。
- [ ] 保留内容标题和封面标题中的人工换行。
- [ ] 逐字稿普通换行只作为段落；`|` 作为人工建议断点。

### 首页交互清单

- [ ] 单视频支持上传或拖入一份逐字稿。
- [ ] 多视频为每条视频分别绑定逐字稿，不能依赖上传顺序猜测。
- [ ] 上传后展示解析结果：内容标题、封面标题、正文长度和警告。
- [ ] 用户可以在提交前分别修改内容标题和封面标题。
- [ ] 缺少内容标题时使用 Markdown 一级标题或文件名。
- [ ] 缺少封面标题时生成建议，但不覆盖用户明确输入。
- [ ] 内容标题与封面标题使用独立字段、独立样式和独立换行结果。

### 测试清单

- [ ] 使用现有真实 Markdown 文档验证回退到“原始文案”。
- [ ] 验证有真实“最终文案”时优先使用最终文案。
- [ ] 验证额外备注和修改记录不会进入 ASR 对齐。
- [ ] 验证标题人工换行进入预览和最终渲染。
- [ ] 验证多视频与逐字稿一一对应。
- [ ] 验证乱码、非 UTF-8、空正文和重复章节的错误提示。

### 阶段出口

- [ ] 用户能在开始任务前明确看到系统将使用的两个标题和逐字稿正文。

## 阶段 3：逐字稿与 ASR 逐字时间对齐

### 目标

使用逐字稿纠正 ASR 错字，同时保留真实音频的逐字时间和脱稿内容。

### 对齐策略

- [ ] 将逐字稿标准化，但保留英文大小写、数字、品牌名和专有词。
- [ ] 将 ASR token 与逐字稿字符或词组进行全局序列对齐。
- [ ] 记录匹配、替换、ASR 新增、稿件缺失和低置信度区间。
- [ ] 高置信度对齐使用逐字稿文字覆盖 ASR 文字。
- [ ] 音频未说出的稿件内容不能自动插入字幕。
- [ ] 音频中的脱稿内容保留，并标记为“稿外口播”。
- [ ] 新增或替换后的文字继承对应 ASR token 时间。
- [ ] 一个 ASR token 对应多个稿件字符时，按词内时长合理分配时间。
- [ ] 无法可靠对齐的区域保留 ASR 原文并生成检查标记。
- [ ] 专有词表从逐字稿和用户词库中提取，并用于后续任务。

### 诊断记录

- [ ] 保存原始 ASR 文本。
- [ ] 保存规范化逐字稿。
- [ ] 保存逐 token 对齐结果。
- [ ] 保存每次替换的原文、结果、置信度和来源。
- [ ] 保存整体覆盖率、替换数、稿外口播数和未对齐区间。

### 测试清单

- [ ] `ai货客系统` 可以根据逐字稿纠正为 `AI获客系统`。
- [ ] 用户没有完全照稿念时不会把未说内容加入字幕。
- [ ] 中英文混合、数字、单位和品牌名不会被拆坏。
- [ ] 重复起句可以分别保留时间，不因对齐而合并错误。
- [ ] 长视频分块对齐时，块边界不漏字、不重复。

### 阶段出口

- [ ] 测试集中的专有词和明确 ASR 错字可以稳定纠正。
- [ ] 纠错不会改变真实口播顺序或凭空增加稿件内容。

## 阶段 4：受约束的语义断句

### 目标

字幕保持预设字号和单行显示，系统先计算全部合法候选，再由 DeepSeek 选择语义最自然的组合。

### 候选生成清单

- [ ] 使用当前视频画幅和具体字幕预设计算真实安全宽度。
- [ ] 计算字体、字号、粗斜体、缩放、字间距、描边、阴影、背景和 X 位置。
- [ ] 使用真实字体测量；临界候选使用 libass 复核。
- [ ] 将逐字稿标点、段落、`|` 和 ASR 停顿转换为候选断点。
- [ ] 加入主谓宾、宾语结束、介词短语结束和完整短语边界候选。
- [ ] 保护否定结构、固定搭配、数字单位、英文单词、品牌名和专有词。
- [ ] 过短句、孤立一两个字和语义残缺候选增加惩罚。
- [ ] 超过最大宽度的候选不能发送给 AI。

### AI 决策清单

- [ ] AI 只能从候选 ID 中选择，不能创建文本或 token。
- [ ] 所选候选必须连续覆盖全部 token。
- [ ] AI 返回后执行覆盖、顺序、宽度和禁断词组校验。
- [ ] 返回非法时重试一次；再次失败使用本地动态规划。
- [ ] AI 不参与删字，断句与内容删减使用不同决策类型。
- [ ] 提示词中加入真实正例和反例，但不依赖提示词保证完整性。

### 测试清单

- [ ] `你买再好的工具｜AI读不到你们业务的上下文` 可以自然断开。
- [ ] `AI读不到你们业务的上下文｜也无法替你做出正确判断` 可以自然断开。
- [ ] 不出现 `生意反而做起来不｜是他们突然变勇敢是` 一类串句。
- [ ] 切换不同预设和画幅后，断句容量随真实宽度变化。
- [ ] 字幕不自动缩小、不自动换成两行、不超出安全区域。
- [ ] 人工断点优先，但不允许制造空字幕或超宽字幕。

### 阶段出口

- [ ] 测试集断句可接受率达到产品内部设定目标。
- [ ] 所有 AI 断句结果都通过确定性校验。

## 阶段 5：自动口误、重复与冗余删除

### 目标

在不依赖人工逐条确认的前提下，自动删除证据充分的卡壳、半句重说和重复内容，并控制误删风险。

### 决策分类

- [ ] `stutter`：连续字词卡壳。
- [ ] `false_start`：说到一半放弃并重新起句。
- [ ] `exact_repeat`：完全重复。
- [ ] `semantic_repeat`：语义重复但措辞不同。
- [ ] `filler`：无意义填充词。
- [ ] `redundant`：不影响信息的冗余表达。

### 自动化策略

- [ ] 标准模式自动处理卡壳、明确半句重说和完全重复。
- [ ] 语义重复必须同时满足文本证据、时间连续性和删除后语法完整。
- [ ] 对比、强调、总结和行动指令默认受保护。
- [ ] 数字、品牌、产品、客户案例和关键结论默认受保护。
- [ ] 删除后重新检查相邻 token 是否构成自然句子。
- [ ] 每个删除操作保存 token、时间、类型、置信度和理由。
- [ ] 精修台可以撤销 AI 删除，撤销结果进入训练反馈。

### 测试清单

- [ ] 正确删除 `我们在广……我们在广西做获客系统` 的前半句。
- [ ] 正确删除 `我们有五个……我们有个做五金加工的客户` 的放弃片段。
- [ ] 不删除 `播放量不是重点，重点是客户` 的对比表达。
- [ ] 删除后视频、字幕、逐字高亮和时间线保持同步。
- [ ] 统计漏删率和错误删除率，错误删除率优先控制。

### 阶段出口

- [ ] 自动删除结果在内部测试集中达到可接受的误删率。
- [ ] 所有删除都可追踪、可撤销、可用于反馈学习。

## 阶段 11：小范围商业测试与持续学习

### 发布策略

- [ ] 先邀请少量真实自媒体用户，不直接全面公开。
- [ ] 每周复盘失败任务、错误删除、断句返工和标题修改。
- [ ] 固定模型、提示词和算法版本，避免无法解释的线上漂移。
- [ ] 新版本先运行离线测试集，再灰度到部分用户。
- [ ] 发现错误删除率上升时可以快速回滚。

### 是否微调模型的判断条件

- [ ] 已积累数量足够、标注一致的真实用户修改数据。
- [ ] 相同错误模式持续出现，提示词和规则无法稳定解决。
- [ ] 已有独立训练集、验证集和不可见测试集。
- [ ] 微调收益能够覆盖成本、部署和版本维护复杂度。
- [ ] 用户已授权对应数据用于训练。

在满足以上条件之前，优先使用：确定性校验、候选约束、真实示例提示和用户词库，不急于微调。
